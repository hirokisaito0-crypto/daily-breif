# 運用を固める（コスト・失敗に気づく・毎朝の見る順番）

## このページの読み方（新卒・非エンジニア向け）

毎朝の自動ビルドは **うまくいっているときは静か**で、失敗しても気づかないと「昨日から止まっている」ままになります。ここでは **お金の暴走防止**・**赤い実行に気づく**・**確認の順番**をまとめます。

### 用語ミニ辞典

| 用語 | ひとことで |
|------|------------|
| **API（エーピーアイ）** | アプリやスクリプトが **プログラムのままサービスを呼ぶ窓口**。ブラウザで ChatGPT を開くのとは別の「プログラム用の入口」です。 |
| **ChatGPT の月額** | ブラウザ／アプリで使う **個人向けプラン**に近いイメージ。 |
| **OpenAI API の課金** | **コードや GitHub Actions から OpenAI を呼ぶとき**の従量課金。上とは **別の請求**になります。 |
| **Usage / Billing** | **いくら使ったか（Usage）**と **請求・上限（Billing）**の画面。公式の表示名は変わることがあります。 |
| **GitHub Actions** | リポジトリで決めたタイミングで **自動でコマンドを実行する仕組み**。成功は緑・失敗は赤で表示されます。 |
| **ワークフロー（workflow）** | 「いつ・何を実行するか」の **レシピファイル**（`.github/workflows/*.yml`）。 |
| **Webhook** | あるサービスから **決まった URL に自動で通知を送る仕組み**。Slack 連携などで使います。 |
| **Ctrl+F5（スーパーリロード）** | ブラウザが **古いページをキャッシュから表示するのを避けて再取得**する操作（環境によっては Shift+再読み込みなど）。 |

---

## 1. OpenAI の使いすぎ防止（API の課金）

**ChatGPT の月額**と **API（プログラムから呼ぶ方）**は別の課金です。日次ブリーフは **API** を使います。

### 1-1. まず見る場所

1. ブラウザで [OpenAI Platform](https://platform.openai.com/) にログイン  
2. **Settings**（またはアカウント）→ **Billing** / **Limits** / **Usage** など（表示名は変わることがあります）

### 1-2. 設定したいもの（目安）

| 目的 | やることのイメージ |
|------|---------------------|
| 月の上限 | **Monthly budget** / **Hard limit** など「これ以上は止める」に近い設定があれば利用 |
| 異常検知 | **Usage** で日別・モデル別の利用量を見る。急増したらフィード件数やモデルを見直す |
| アラート | メール通知やしきい値アラートがあればオン（公式UIは更新されるので、画面に出ている説明を優先） |

### 1-3. このプロジェクト側でコストを抑えるコツ

- **フィード数**：[`brief_pipeline/config/feeds.yaml`](../brief_pipeline/config/feeds.yaml) の失敗・不要なURLを減らす（取得失敗のリトライが減る）。  
- **モデル**：[`brief_pipeline/build.py`](../brief_pipeline/build.py) の `--model`（既定 `gpt-4o-mini`）を、公式が推す安価モデルに合わせる。  
- **頻度**：すでに **ロンドン7時に1日1回**なので、追加で手動 Run しすぎない。

## 2. 失敗に気づく（GitHub Actions が赤）

### 2-1. いちばん簡単：GitHub がメールを送る設定（無料・追加コード不要）

あなたの **GitHub アカウント**の通知設定で、Actions の失敗をメールにできます。

1. GitHub にログイン  
2. 右上のアイコン → **Settings**（ユーザー設定）  
3. 左メニュー **Notifications**  
4. **Actions** の項目で、**失敗したワークフロー**に関する通知をオンにする  

表示は英語のことが多いですが、「failed workflows」「Actions failures」などにチェックを入れるイメージです。

これで **Daily brief が赤になったとき**にメールが届くようにできます（設定次第）。

### 2-2. リポジトリを「ウォッチ」する

リポジトリページ → 上部の **Watch** → **All Activity** など、通知レベルを上げると、見逃しにくくなります（通知が増えるので好みで）。

### 2-3. Slack / Teams（あとでやるなら）

Webhook URL を GitHub Secrets に入れて、失敗時だけ投稿するワークフローを足す方法があります。**まずはメール通知で十分**なことが多いです。

## 3. 毎朝の「見る順番」の習慣（おすすめ）

公開サイトを開く前に、次の順がおすすめです。

1. **GitHub → Actions**  
   - **Daily brief** の **最新の実行が緑（成功）か**  
   - 赤なら **ログを開く**（前の成功版がサイトに残っていることもある）  
2. **問題なければ** Cloudflare / GitHub Pages の URL を開く（必要なら **Ctrl+F5** で再読み込み）

### 心のメモ

- **緑＝ビルドとデプロイは通った**が、**中身が期待どおりか**は別問題（RSS が0件など）。長い目でログの `ingested ...` もたまに見ると安心です。

## 4. チェックリスト（印刷してもよい）

毎週1回でよいので：

- [ ] OpenAI の **Usage / Billing** をざっと見た  
- [ ] GitHub Actions の **Daily brief** が直近で緑続きか見た  
- [ ] 公開URLを開いたとき **Ctrl+F5** した（キャッシュ対策）

## 5. トラブル時に見る場所

| 症状 | 見る場所 |
|------|-----------|
| サイトが古い | Actions が赤／スキップされていないか → `Build brief` ログ |
| お金が増えた | OpenAI Usage → 同日に手動 Run を何度もしていないか |
| Cloudflare だけ更新されない | `CLOUDFLARE_*` の Secrets と Wrangler のログ |
